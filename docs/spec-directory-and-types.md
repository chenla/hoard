# Hoard Directory Conventions and Card Type System

**Status:** Working Draft, 2026-05-01
**Audience:** Implementors building tools that read or write hord repositories

---

## 1. Conformance

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHOULD",
"SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document
are to be interpreted as described in RFC 2119.

A **hord** is a directory tree managed by Git that contains a `.hord/`
metadata overlay.  A **card** is a single file (org-mode or Markdown)
representing one entity in the hord.  A **compiler** is any program
that reads card files and produces quads and index entries in `.hord/`.

---

## 2. Repository Structure

A conforming hord repository has the following layout:

```
<hord-root>/
  .hord/
    config.toml
    index.tsv
    vocab/
      terms.tsv
      relations.tsv
    quads/
      <prefix>/
        <uuid>.tsv
  content/
    <card files>
  capture/
    <capture card files>
  lib/
    blob/
      <binary files>
```

The hord root is defined as the nearest ancestor directory that
contains a `.hord/` subdirectory.  The hord root MUST also be the
root of a Git repository (i.e., the same directory that contains
`.git/`).

---

## 3. The `.hord/` Directory

The `.hord/` directory is the metadata overlay.  It is created by
`hord init` and MUST NOT be edited by hand except for `config.toml`.
Its contents are derived data, regenerated by `hord compile`.

### 3.1. `config.toml`

The hord configuration file.  Format:

```toml
[hord]
name = "tps-demo"
version = "0.1.0"

[vocab]
source = "local"
```

The `[hord]` table is REQUIRED and MUST contain:

| Key       | Type   | Description                                         |
|-----------|--------|-----------------------------------------------------|
| `name`    | string | Human-readable name for this hord.  Defaults to the directory name if omitted during `hord init`. |
| `version` | string | Hoard format version this hord was created with.    |

The `[vocab]` table is REQUIRED and MUST contain:

| Key      | Type   | Description                                          |
|----------|--------|------------------------------------------------------|
| `source` | string | Vocabulary source.  Currently only `"local"` is defined, meaning vocabulary files are read from `.hord/vocab/`. |

Implementations MAY define additional tables and keys.  Unknown keys
MUST be preserved when rewriting `config.toml`.

### 3.2. `index.tsv`

A tab-separated file mapping relative file paths to UUIDs.  The
compiler writes this file; it MUST NOT be edited by hand.

The first line is a header:

```
path	uuid
```

Each subsequent line maps one card file to its UUID:

```
content/Kanban--4.org	d4e5f6a7-1001-4000-8000-000000000011
content/Taiichi_Ohno--7.org	e5f6a7b8-2001-4000-8000-000000000021
```

Paths are relative to the hord root.  Entries are sorted
lexicographically by path.  The compiler MUST overwrite this file
completely on each compilation (not append).

### 3.3. `vocab/`

Contains the vocabulary definition files that govern the type system
and relation predicates.  `hord init` copies the default vocabulary
into this directory.

**`vocab/terms.tsv`** defines all valid term identifiers:

```
id	label	scope_note
wh:con	Concept	Word-hord: concept record
wh:pat	Pattern	Word-hord: proto-pattern record
v:bt	BT	Broader term (ANSI/NISO Z39.19)
v:title	TITLE	Display title of the entity
```

The `id` column is the canonical identifier used in quads and property
drawers.  The `wh:` prefix denotes card types (Section 5).  The `v:`
prefix denotes relation predicates and metadata properties.

**`vocab/relations.tsv`** defines relationships between vocabulary
terms themselves (e.g., that BT and NT are inverses):

```
subject	predicate	object	note
v:bt	v:rt	v:nt	BT and NT are inverse thesaurus relations
v:uf	v:rt	v:use	UF and USE are inverse preference relations
```

### 3.4. `quads/`

Contains compiled quad files, sharded into subdirectories by UUID
prefix.  Each card's quads are stored in a single TSV file named by
the card's UUID.

The sharding scheme uses the first four characters of the UUID as the
subdirectory name:

```
.hord/quads/d4e5/d4e5f6a7-1001-4000-8000-000000000011.tsv
.hord/quads/852a/852a6e49-4b9c-429e-b612-6c505ab78827.tsv
```

Each quad file has the header:

```
subject	predicate	object	context
```

Followed by one quad per line:

```
d4e5f6a7-1001-4000-8000-000000000011	v:type	wh:con	38cbc45a0fd12092ccd8368c10ba17a0bb903d9b
d4e5f6a7-1001-4000-8000-000000000011	v:title	Kanban---4	38cbc45a0fd12092ccd8368c10ba17a0bb903d9b
d4e5f6a7-1001-4000-8000-000000000011	v:bt	c348132e-7cdf-438d-9a1f-69d75f382bee	38cbc45a0fd12092ccd8368c10ba17a0bb903d9b
d4e5f6a7-1001-4000-8000-000000000011	v:uf	Signboard system	38cbc45a0fd12092ccd8368c10ba17a0bb903d9b
```

The four fields are:

| Field       | Description                                              |
|-------------|----------------------------------------------------------|
| `subject`   | UUID of the entity this quad describes.                  |
| `predicate` | Vocabulary term ID for the relationship (e.g., `v:bt`).  |
| `object`    | Target UUID, vocabulary term ID, or literal string.      |
| `context`   | Git blob hash of the source file at compile time.  Provides provenance: which version of the source file produced this quad. |

The compiler MUST overwrite a card's quad file completely on each
compilation.

## 4. Content Directories

### 4.1. `content/`

The `content/` directory holds **reference cards** --- the curated,
permanent knowledge base.  All card types except `wh:cap` default to
this directory when created via `hord new`.

There is no enforced subdirectory structure within `content/`.
Implementations MAY organize cards into subdirectories; the compiler
scans recursively.

### 4.2. `capture/`

The `capture/` directory holds **capture cards** (`wh:cap`) --- quick
notes, observations, fleeting thoughts, and raw material that has not
yet been processed into reference cards.

When `hord new` creates a card of type `wh:cap`, it defaults the
output directory to `capture/` rather than `content/`.  This is a
convention, not a hard constraint: the compiler scans both directories
(and any others) recursively.

The capture/content split exists to support a workflow where raw
material flows in through `capture/` and is gradually refined into
permanent cards in `content/`.

### 4.3. `lib/blob/`

The `lib/blob/` directory stores binary files associated with the
hord --- PDFs, images, data files, and other non-card content.  These
files are tracked by Git but are not parsed by the compiler.

Cards reference blob files using relative paths from the hord root:

```
:SOURCE:    lib/blob/ohno-1988-tps.pdf
```

There is no enforced naming convention for blob files.  Implementors
SHOULD use descriptive filenames.  Large binary files SHOULD be
managed with Git LFS.

## 5. File Naming Convention

Card filenames encode the entity's title and type using this pattern:

```
<Slug>--<suffix>.<ext>
```

Where:

| Component  | Description                                              |
|------------|----------------------------------------------------------|
| `<Slug>`   | Title converted to a filename-safe slug.  Spaces become underscores.  Non-alphanumeric characters (except hyphens and underscores) are stripped.  Multiple consecutive underscores are collapsed to one. |
| `--`       | Two ASCII hyphens.  This is the separator between slug and suffix.  It MUST be two hyphens (`--`), not an em dash. |
| `<suffix>` | Numeric type code (see Section 6.2).                     |
| `<ext>`    | File extension: `org` for org-mode, `md` for Markdown.   |

Examples:

```
Kanban--4.org                 (concept)
Taiichi_Ohno--7.org           (person)
The_Machine_That_Changed_the_World--6.org   (work)
Toyota_Production_System--4.org             (concept)
Concept--8.org                (category)
```

The slug is generated by the `slugify()` function:

1. Replace spaces with underscores.
2. Remove all characters that are not ASCII word characters or hyphens.
3. Collapse runs of underscores to a single underscore.
4. Strip leading and trailing underscores.

### 5.1. Display Title vs. Filename

The display title (used in `#+TITLE:`, headings, and PT relations)
uses an em dash between the title and suffix:

```
Kanban---4
```

The filename uses two ASCII hyphens:

```
Kanban--4.org
```

This distinction matters.  Filenames use ASCII for portability; display
titles use the typographically correct em dash for readability.

## 6. Card Types

### 6.1. The `wh:` Namespace

All card types are defined in the `wh:` (word-hord) namespace of the
vocabulary.  The `wh:` prefix identifies entity classification terms
as distinct from relation predicates (`v:` prefix) and Strata terms
(`v:s-` prefix).

Each card type has three representations:

| Representation  | Example     | Used in                                     |
|-----------------|-------------|---------------------------------------------|
| Vocab ID        | `wh:con`    | Property drawers, YAML frontmatter, quads   |
| Numeric suffix  | `4`         | Filenames, display titles                   |
| Human label     | `Concept`   | Vocabulary scope notes, documentation       |

### 6.2. Type Table

The following table lists all defined card types, ordered by numeric
suffix.

| Suffix | Vocab ID  | Label        | Purpose                                          |
|--------|-----------|--------------|--------------------------------------------------|
| 3      | `wh:pat`  | Pattern      | A proto-pattern or design pattern.  Describes a recurring solution to a problem in context. |
| 4      | `wh:con`  | Concept      | A concept or idea.  The default and most common card type.  Represents a single well-defined notion. |
| 5      | `wh:key`  | Keystone     | A keystone or expression record.  A concept that has been developed into a full argument or essay-length treatment. |
| 6      | `wh:wrk`  | Work         | A bibliographic work.  Books, papers, articles, films, or any citable creative work. |
| 7      | `wh:per`  | Person       | A person or named individual.  Biographical and relational information. |
| 8      | `wh:cat`  | Category     | A category or top-level classification term.  Used to organize the thesaurus hierarchy. |
| 9      | `wh:sys`  | System       | A system or meta record.  Used for records about the hord itself or its organizational infrastructure. |
| 10     | `wh:pla`  | Place        | A geographic place or location.                  |
| 11     | `wh:evt`  | Event        | A historical or scheduled event.                 |
| 12     | `wh:obj`  | Object       | A physical or conceptual object (artifact, tool, specimen). |
| 13     | `wh:org`  | Organization | An organization, institution, company, or group. |
| 14     | `wh:cap`  | Capture      | A capture card.  Quick notes, observations, fleeting thoughts.  Raw material that has not been processed into a reference card.  Defaults to `capture/` directory. |

Suffix numbers 1 and 2 are reserved.  Implementations MUST NOT assign
types to these suffixes.

### 6.3. Type Shortcuts

The `hord new` command accepts short names for types.  Both the
three-letter code and the full English word are accepted:

```
hord new "Kanban" -t con         # wh:con
hord new "Kanban" -t concept     # wh:con
hord new "Kanban" -t wh:con      # wh:con (explicit)
hord new "Taiichi Ohno" -t per   # wh:per
hord new "My Book" -t wrk        # wh:wrk
```

The resolution order is:

1. If the value starts with `wh:`, use it as-is.
2. Otherwise, look up the value (case-insensitive) in the shortcuts
   table.
3. If not found, report an error.

### 6.4. Required Fields by Type

All card types share a common set of REQUIRED properties:

| Property   | Description                                              |
|------------|----------------------------------------------------------|
| `ID`       | A UUID v4 identifying this card.  Generated by `hord new`.  MUST be globally unique. |
| `TYPE`     | The vocab ID of the card type (e.g., `wh:con`).         |
| `CREATED`  | Timestamp in the format `YYYY-MM-DDTHH:MM`, optionally followed by `@Location`. |

The following properties are OPTIONAL for all types:

| Property       | Description                                          |
|----------------|------------------------------------------------------|
| `GEO`          | Geographic coordinates in MGRS or decimal format.    |
| `LICENCE`      | License under which the card content is released.    |
| `AUTHOR`       | Author of the card or the work it describes.         |
| `ROAM_ALIASES` | Space-separated, quoted alternate names.  Each alias generates a `v:uf` quad. |

**Work cards (`wh:wrk`)** SHOULD include a `** Bibliographic Data`
section with its own property drawer containing:

| Property    | Description                                             |
|-------------|---------------------------------------------------------|
| `BIB-TYPE`  | Bibliographic type (e.g., `book`, `article`, `inproceedings`). |
| `AUTHOR`    | Author(s), semicolon-separated for multiple.            |
| `TITLE`     | Title of the work.                                      |
| `YEAR`      | Publication year.                                       |
| `PUBLISHER` | Publisher name.                                         |
| `CITEKEY`   | Citation key (e.g., `ohno:1988tps`).                    |

## 7. Card File Format

A card file is either an org-mode file (`.org`) or a Markdown file
(`.md`).  Both formats encode the same metadata; the compiler
normalizes them into identical quads.

### 7.1. Org-mode Format

```org
#   -*- mode: org; fill-column: 60 -*-
#+STARTUP: showall
#+TITLE:   Kanban---4
#+FILETAGS: "hord" "concept"

* Kanban---4
  :PROPERTIES:
  :ID:        d4e5f6a7-1001-4000-8000-000000000011
  :TYPE:      wh:con
  :CREATED:   2026-04-22T10:00@Hong Kong
  :LICENCE:   MIT/CC BY-SA 4.0
  :END:

** Relations
   - TT :: [[id:852a6e49-4b9c-429e-b612-6c505ab78827][Concept---8]]
   - BT :: [[id:c348132e-7cdf-438d-9a1f-69d75f382bee][Toyota Production System---4]]
   - RT :: [[id:d4e5f6a7-1003-4000-8000-000000000013][Just-in-Time Manufacturing---4]]
   - UF :: Signboard system
   - PT :: Kanban---4

** Notes

Content goes here.
```

Structure:

1. **File-level directives**: `#+TITLE:` and `#+FILETAGS:` at the top.
2. **Top-level heading**: The card title with type suffix (em dash).
3. **Property drawer**: Metadata fields under the top-level heading.
4. **Relations section**: A `** Relations` subheading containing
   thesaurus relationships as list items in the form `- REL :: target`.
5. **Notes section**: Free-form content.

Relations use org-mode ID links for cross-references:

```org
- BT :: [[id:<uuid>][Display Title]]
```

For labels without a target UUID (PT, UF entries that are plain text):

```org
- UF :: Signboard system
- PT :: Kanban---4
```

### 7.2. Markdown Format

```markdown
---
id: d4e5f6a7-1001-4000-8000-000000000011
type: wh:con
title: Kanban---4
created: 2026-04-22T10:00@Hong Kong
license: MIT/CC BY-SA 4.0
relations:
  - "TT: 852a6e49-4b9c-429e-b612-6c505ab78827  # Concept---8"
  - "BT: c348132e-7cdf-438d-9a1f-69d75f382bee  # Toyota Production System---4"
  - "UF: Signboard system"
  - "PT: Kanban---4"
aliases:
  - "Signboard system"
---

# Kanban---4

Content goes here.
```

Metadata is encoded in YAML frontmatter (delimited by `---`).
Relations are a YAML list of strings in the form `"REL: target"`.
When the target is a UUID, an inline comment after `#` provides the
human-readable label.

## 8. Type Inference

The compiler determines a card's type through a three-step fallback
chain.  The first source that yields a result wins.

### 8.1. Step 1: Property Drawer / Frontmatter

The compiler reads the `TYPE` property (org-mode) or `type` field
(Markdown frontmatter).  If present and non-empty, this is the
authoritative type.

Both old-style and new-style type values are accepted:

| Old-style value    | Resolved to |
|--------------------|-------------|
| `con/concept`      | `wh:con`    |
| `concept`          | `wh:con`    |
| `pat/pattern`      | `wh:pat`    |
| `per/person`       | `wh:per`    |
| `wrk/work`         | `wh:wrk`    |
| `wh:con`           | `wh:con`    |

Values already in `wh:` form are used as-is.  Old-style values are
mapped through `OLD_TYPE_MAP`.

### 8.2. Step 2: Filename Suffix

If the property drawer does not contain a TYPE, the compiler extracts
the numeric suffix from the filename.

The compiler matches the pattern `--<digits>` at the end of the
basename (before the extension).  Both ASCII double-hyphen (`--`) and
Unicode em dash are recognized:

```
Kanban--4.org     -> suffix "4" -> wh:con
Taiichi_Ohno--7.org -> suffix "7" -> wh:per
```

The suffix-to-type mapping:

| Suffix | Type      |
|--------|-----------|
| `3`    | `wh:pat`  |
| `4`    | `wh:con`  |
| `5`    | `wh:key`  |
| `6`    | `wh:wrk`  |
| `7`    | `wh:per`  |
| `8`    | `wh:cat`  |
| `9`    | `wh:sys`  |
| `10`   | `wh:pla`  |
| `11`   | `wh:evt`  |
| `12`   | `wh:obj`  |
| `13`   | `wh:org`  |

Note: suffix `14` (`wh:cap`) is not in the filename inference map
because capture cards are identified by their property drawer, not
by filename convention alone.

### 8.3. Step 3: Filetags

If neither the property drawer nor the filename provides a type, the
compiler checks the `#+FILETAGS:` directive (org-mode) or `tags`
field (Markdown).

Filetags are matched case-insensitively against the old-style type
names:

```org
#+FILETAGS: "hord" "concept"
```

The tag `"concept"` matches `wh:con`.

This is a last-resort fallback.  Cards SHOULD always have an explicit
TYPE property.

### 8.4. Inference Precedence Summary

```
Property drawer TYPE  -->  Filename suffix  -->  Filetags
    (highest)                                    (lowest)
```

If none of the three sources yields a type, the card is still compiled
(its UUID and relations are extracted), but no `v:type` quad is
emitted.

## 9. Relation Types

Relations express thesaurus-style relationships between cards,
following ANSI/NISO Z39.19.

### 9.1. Thesaurus Relations

| Code | Vocab ID | Meaning                    | Target          |
|------|----------|----------------------------|-----------------|
| TT   | `v:tt`   | Top term                   | UUID (link)     |
| PT   | `v:pt`   | Preferred term (self-label)| Literal string  |
| BT   | `v:bt`   | Broader term               | UUID (link)     |
| BTG  | `v:btg`  | Broader term, generic      | UUID (link)     |
| BTI  | `v:bti`  | Broader term, instance     | UUID (link)     |
| BTP  | `v:btp`  | Broader term, partitive    | UUID (link)     |
| NT   | `v:nt`   | Narrower term              | UUID (link)     |
| NTG  | `v:ntg`  | Narrower term, generic     | UUID (link)     |
| NTI  | `v:nti`  | Narrower term, instance    | UUID (link)     |
| NTP  | `v:ntp`  | Narrower term, partitive   | UUID (link)     |
| RT   | `v:rt`   | Related term               | UUID (link)     |
| UF   | `v:uf`   | Used for (non-preferred)   | Literal string  |
| USE  | `v:use`  | Use instead (preferred)    | UUID (link)     |

### 9.2. Strata Relations

| Code | Vocab ID    | Meaning                        | Target      |
|------|-------------|--------------------------------|-------------|
| WO   | `v:s-wo`    | Is a Whole                     | UUID (link) |
| EO   | `v:s-eo`    | Is an Expression of a Whole    | UUID (link) |
| MO   | `v:s-mo`    | Is a Manifestation of an Expression | UUID (link) |
| IO   | `v:s-io`    | Is an Instance of a Manifestation  | UUID (link) |

### 9.3. Relation Compilation

The compiler maps relation codes to vocabulary predicates using the
`REL_TO_PREDICATE` table.  For each relation line in a card:

1. Extract the relation type code (e.g., `BT`).
2. Look up the predicate in `REL_TO_PREDICATE`.  If unknown, emit a
   warning and skip.
3. If the target is an org-mode ID link (`[[id:<uuid>][label]]`),
   use the UUID as the object.
4. If the target is a plain string (no link), use the string as the
   object.

## 10. Compilation Output

When `hord compile` runs, it performs the following:

1. Recursively scans the specified path for `.org` and `.md` files.
2. Parses each file to extract metadata (UUID, type, title, author,
   relations, aliases).
3. For each valid card (one that has a UUID), emits quads:
   - One `v:type` quad (if type is known).
   - One `v:title` quad (if title is present).
   - One `v:author` quad (if author is present).
   - One quad per relation.
   - One `v:uf` quad per alias from `ROAM_ALIASES`.
4. Writes each card's quads to `.hord/quads/<prefix>/<uuid>.tsv`.
5. Overwrites `.hord/index.tsv` with the complete path-to-UUID
   mapping.

The context field of every quad is the Git blob hash of the source
file, computed via `git hash-object`.  This provides immutable
provenance: given a quad, you can identify exactly which version of
the source file produced it.

## 11. Examples

### 11.1. Creating a New Hord

```
$ mkdir my-hord && cd my-hord
$ git init
$ hord init --name "My Knowledge Base"
Initialized .hord/ in /home/user/my-hord
  name: My Knowledge Base
  config: /home/user/my-hord/.hord/config.toml
  vocab: /home/user/my-hord/.hord/vocab/
  quads: /home/user/my-hord/.hord/quads/
```

### 11.2. Creating Cards

```
$ hord new "Kanban" -t con
Created content/Kanban--4.org
  UUID: a7b8c9d0-...
  Type: wh:con

$ hord new "Taiichi Ohno" -t per
Created content/Taiichi_Ohno--7.org
  UUID: b8c9d0e1-...
  Type: wh:per

$ hord new "Quick thought about flow" -t cap
Created capture/Quick_thought_about_flow--14.org
  UUID: c9d0e1f2-...
  Type: wh:cap

$ hord new "My Book" -t wrk -f md
Created content/My_Book--6.md
  UUID: d0e1f2a3-...
  Type: wh:wrk
```

### 11.3. Compiling

```
$ hord compile
Compiled 24 files -> 127 quads
Index: 24 entries
```

### 11.4. Resulting Index

```
path	uuid
capture/Quick_thought_about_flow--14.org	c9d0e1f2-...
content/Kanban--4.org	a7b8c9d0-...
content/My_Book--6.md	d0e1f2a3-...
content/Taiichi_Ohno--7.org	b8c9d0e1-...
```

## Appendix A. Grammar

The following ABNF describes the filename convention:

```abnf
card-filename = slug "--" suffix "." extension
slug          = 1*(ALPHA / DIGIT / "_" / "-")
suffix        = 1*DIGIT
extension     = "org" / "md"
```

The following ABNF describes the org-mode relation line:

```abnf
relation-line = SP "-" SP rel-code SP "::" SP target
rel-code      = "TT" / "PT" / "BT" / "BTG" / "BTI" / "BTP"
              / "NT" / "NTG" / "NTI" / "NTP"
              / "RT" / "UF" / "USE"
              / "WO" / "EO" / "MO" / "IO"
target        = id-link / literal
id-link       = "[[id:" uuid "][" label "]]"
literal       = 1*(%x20-7E / UTF8-char)
```

## Appendix B. Vocabulary Namespaces

| Prefix | Namespace   | Purpose                                    |
|--------|-------------|--------------------------------------------|
| `wh:`  | Word-hord   | Card type classification terms             |
| `v:`   | Vocabulary  | Relation predicates and metadata properties|
| `v:s-` | Strata      | WEMI entity-relationship predicates        |
